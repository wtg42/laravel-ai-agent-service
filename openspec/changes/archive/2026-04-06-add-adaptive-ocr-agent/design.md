## Context

現有系統的 AI 能力都以純文字輸入為中心，`ChineseNameDetectionAgent` 與 `EmailScanAgent` 都假設上游已提供可直接分析的文字內容。這使系統無法處理來自截圖、手機拍照文件或掃描圖檔的實際輸入，也缺少將圖片轉為文字後再銜接既有個資掃描流程的能力。

本次變更的核心難點不只是「讀圖」，而是不同圖片品質問題需要不同修正策略：有些圖片需要旋轉，有些需要提高對比或裁切，有些則應直接分析。若將這些步驟寫成固定流程，容易造成過度處理、效能浪費或錯誤修正。因此本設計改採 LLM 作為決策中心，由 agent 先判斷下一步，再由應用層執行工具並重新附圖進入下一輪。

此外，Laravel AI SDK 提供 agent、tools、attachments 與 structured output 等 building blocks，但未提供完整的多輪影像工具鏈 orchestration。因此需要在 Laravel 應用層明確管理工具執行、重試上限、處理後圖片保存與最終 OCR 文字輸出。

## Goals / Non-Goals

**Goals:**
- 提供單張圖片輸入的 `AdaptiveOcrAgent`，由模型決定是否需要影像前處理。
- 讓 agent 能在有限步數內選擇旋轉、銳化、提高對比、裁切或直接開始辨識。
- 在 Laravel 應用層建立可觀測的 re-prompt loop，讓每次工具輸出都能重新附圖給 agent。
- 產出穩定的 OCR 文字結果，供後續姓名與個資掃描流程重用。
- 保留每輪決策與處理後圖片，方便除錯與品質驗證。

**Non-Goals:**
- 不在第一版處理多頁文件、PDF 或批次圖片輸入。
- 不在第一版直接輸出完整個資掃描結果；下游仍由既有文字型 agent 或後續專用 agent 處理。
- 不在第一版引入模型微調、訓練資料管理或自動評分機制。
- 不保證所有低品質圖片都可修復；當圖片不可用時允許 agent 明確拒絕分析。

## Decisions

### 1. 使用 `AdaptiveOcrAgent` 作為決策中心，而非固定式前處理流程

系統將新增專用的 `AdaptiveOcrAgent`，先根據附加圖片判斷下一步是呼叫工具還是直接分析內容。agent 的每輪輸出採 structured output，至少包含 `action`、`reason`、`confidence` 與必要的參數欄位。可接受的動作包含 `rotate`、`sharpen`、`increase_contrast`、`crop`、`analyze_now` 與 `reject_image`。

選擇此做法是因為圖片品質問題具有高度變異性，由模型根據當前圖片狀態判斷，比寫死順序更容易避免不必要的處理。替代方案是固定 pipeline，例如先灰階、再銳化、再 OCR，但此法難以區分哪些圖片本來就可直接讀取，也難以表示「此圖應直接放棄」的決策。

### 2. 將影像轉換能力設計為可被 agent 呼叫的工具，但由應用層控制迴圈

每個影像處理步驟都以獨立工具表達，例如旋轉、銳化、提高對比與裁切。工具只負責將來源圖片轉成新圖片並回傳結果資訊，例如輸出檔案位置、執行參數與圖片摘要。工具不負責決策，也不直接進入下一輪分析。

Laravel 應用層會讀取 agent 的 `action`，執行對應工具後，再把新圖片重新作為 attachment 傳回同一個 agent 進行下一輪。選擇此做法，是因為 Laravel AI SDK 雖支援 tools 與 attachments，但未明確提供「tool 產出新圖片後在同一回合自動納入模型上下文」的內建影像迴圈能力。替代方案是完全依賴 agent 內部工具鏈，但會使圖片附件狀態與除錯路徑不透明。

### 3. 第一版將工具集合收斂為四種常見文件修正能力

第一版工具僅包含：
- 旋轉圖片
- 銳化圖片
- 提高對比
- 裁切圖片

這四種能力已能覆蓋常見的文件方向錯誤、模糊、低對比與主體區域過大問題。暫不納入更多操作，例如去背、透視校正、多區塊裁切或超解析度，以降低 agent 決策空間與實作複雜度。替代方案是一次加入更完整的影像處理工具箱，但會提高錯誤操作組合與測試矩陣。

### 4. OCR 結果與決策結果分離

當 agent 判定 `analyze_now` 時，系統會進入文字抽取階段並輸出穩定的 OCR 結果。第一版至少需要回傳完整抽取文字，並保留必要的品質資訊，例如是否有低信心或圖片品質警告。下游姓名與個資抽取仍視為獨立能力，透過既有或未來的文字型 agent 處理。

這樣的分層可以把「圖片是否可讀」與「文字代表什麼個資」分開，便於除錯與替換。替代方案是直接讓同一個多模態 agent 從圖片輸出最終個資結果，但會把 OCR 失敗與個資判讀失敗耦合在一起。

### 5. 明確限制迭代上限與工具使用規則

為避免 agent 進入無限旋轉或過度前處理，系統需在應用層施加保護：
- 單次請求最多允許有限輪數的工具處理與 re-prompt。
- 同一工具不可無限制重複。
- 若多輪後仍不可讀，agent 應可回傳 `reject_image`。
- 每輪都必須保留原圖與處理後圖片的關聯，方便追蹤。

替代方案是完全依賴 prompt 約束 agent 自律，但這無法取代應用層保護機制。

## Risks / Trade-offs

- [模型過度處理圖片] → 以應用層限制最大步數、禁止無限重複工具，並在 prompt 中要求優先最小必要處理。
- [小模型 OCR 品質不足] → 第一版保留明確拒絕分析與人工評估空間，必要時後續可切換更大 vision 模型。
- [圖片工具回傳格式過於複雜] → 工具第一版僅回傳新圖片位置與必要 metadata，避免讓 agent 依賴過多工具細節。
- [處理後圖片管理混亂] → 將每輪輸出圖片與對應決策保存為可追蹤 artifact，並建立固定命名與清理策略。
- [單張圖片設計難以直接延伸到多頁文件] → 在 spec 中明確將第一版限制為單張圖片，未來多頁需求再另開能力。

## Migration Plan

1. 新增圖片輸入端點、request 驗證與 `AdaptiveOcrAgent`。
2. 新增第一版影像工具與應用層 orchestration loop。
3. 串接 OCR 結果輸出與既有文字型掃描流程。
4. 以 feature tests 驗證 agent 決策、工具執行、停止條件與 OCR 輸出結構。
5. 若功能需要回滾，可停用圖片端點與 orchestration 入口，不影響既有純文字掃描 API。

## Open Questions

- 第一版 OCR 結果是否只需回傳 `text`，或需同時回傳 `quality_issues`、`warnings` 與局部區塊資訊。
- 圖片工具底層將採用何種實作方式，是否需要額外系統相依或 PHP 影像函式庫。
- `AdaptiveOcrAgent` 是否在第一版就要暴露給外部 API，或先作為內部流程能力。
