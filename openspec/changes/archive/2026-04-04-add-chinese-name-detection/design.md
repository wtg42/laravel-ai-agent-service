## Context

此專案目前已安裝 Laravel AI SDK，並以本地 Ollama + Gemma 4 E2B 作為主要模型方向，但應用層仍停留在 Laravel skeleton 階段，尚未建立任何 `app/Ai/Agents`、`app/Ai/Tools` 或 API 路由。中文姓名偵測將成為第一個端到端落地的 AI 能力，必須同時驗證本地模型整合、結構化輸出、HTTP 介面，以及後續擴充為更完整個資掃描能力的可延展性。

## Goals / Non-Goals

**Goals:**
- 提供一個可接收單段文字並回傳明確中文姓名的 REST API。
- 以 `laravel/ai` Agent 對接 Ollama 模型，並使用結構化輸出固定回應格式。
- 透過 Laravel 端後處理減少稱謂、單位名稱、模糊詞彙造成的常見誤判。
- 建立可測試的第一個 AI 能力垂直切片，供後續 Email 掃描與文件稽核能力重用。

**Non-Goals:**
- 不處理 OCR、影像輸入或文件上傳。
- 不處理其他個資類型，例如身份證號、電話、Email 或統編。
- 不處理遮罩姓名、暱稱、純稱謂或身份推論。
- 不在此變更中建立多 Agent 工作流編排能力。

## Decisions

### 1. 以單一專用 Agent 承接中文姓名抽取
系統將新增一個專門負責中文姓名偵測的 Agent，而不是直接在 controller 內以匿名 agent 呼叫模型。這樣可讓指示、模型設定、結構化 schema 與測試 fake 集中在單一類別中，符合專案 README 所規劃的 `Agent per use case` 架構。

替代方案：使用匿名 agent 快速完成 MVP。此作法雖較快，但會讓 prompt、schema 與測試耦合在控制器層，不利於日後擴充成多個辨識能力。

### 2. 採用「LLM 抽取候選 + Laravel 後處理收斂」
Gemma 4 E2B 小模型負責依語境抽取明確中文姓名候選；Laravel 端負責去重、清理空白與標點、排除明顯非姓名詞彙，以及處理空結果。這符合專案「AI 負責辨識、Tool / 應用層負責規則與穩定性」的設計原則。

替代方案：純規則抽取或純 LLM 直出。純規則在中文姓名語境判斷上召回率不足；純 LLM 則容易在小模型下出現誤判與回應不穩。

### 3. 輸出以 precision-first 為優先，並保留 evidence
第一版只接受明確全名，不要求涵蓋遮罩姓名、稱謂型稱呼或暱稱。每個偵測結果應包含姓名值與支持片段，必要時可附模型信心值。保留 evidence 能提升除錯、誤判分析與未來人工複核體驗。

替代方案：僅回傳姓名字串陣列。這雖然簡潔，但會降低 API 消費端與開發者追查誤判的能力。

### 4. API 採能力導向路由，為後續 PII 擴充保留分組空間
此能力應以獨立 API 路由暴露，並歸入未來可擴充的 PII 分類下。路由命名將以單一文字偵測能力為準，而非直接綁定 Email 場景，避免未來其他文字來源重複實作相同邏輯。

替代方案：直接實作成 Email 掃描端點的一部分。這會讓姓名偵測與更大範圍的 Email 個資掃描高度耦合，不利於單獨測試與逐步擴充。

### 5. 測試以 feature tests 為主，Agent fake 為輔
HTTP 層、輸入驗證與回應格式應透過 feature tests 驗證；Agent 互動則以 `laravel/ai` 提供的 fake 能力固定輸出，確保測試不依賴本地模型服務。另可保留少量手動驗證樣本，用於實際觀察 Gemma 4 E2B 的誤判模式。

替代方案：以真實 Ollama 回應作為主要測試依據。這會使 CI 或本地測試高度依賴外部服務狀態，降低穩定性。

## Risks / Trade-offs

- [小模型誤將稱謂或單位名稱當成人名] → 以 prompt 限制為第一層，Laravel 後處理為第二層，並以負樣本測試案例持續校正。
- [只追求 precision 會降低召回率] → 在 MVP 階段接受部分漏抓，以換取 API 可用性；待收集真實樣本後再評估是否放寬邊界。
- [結構化輸出欄位過多會提高小模型不穩定性] → 第一版將 schema 維持精簡，只保留姓名、證據與必要的信心欄位。
- [本地模型服務未啟動時 API 失敗] → 於應用層明確處理 provider 錯誤，避免回傳不清楚的例外訊息。

## Migration Plan

1. 新增中文姓名偵測 Agent、必要的後處理邏輯與 API 端點。
2. 以 feature tests 和 Agent fake 驗證輸入、輸出與空結果處理。
3. 在本地啟動 Ollama 與 Gemma 4 E2B 後，使用實際文字樣本做手動驗證。
4. 若模型表現不符合預期，可在不變更 API 契約的前提下調整 prompt 與後處理規則。

## Open Questions

- 第一版回應是否需要公開 `confidence`，或僅內部保留給除錯使用。
- 後處理是否需要引入簡單姓氏字典，或先以 stopword / 稱謂過濾維持最小實作。
- API 路由最終應採用能力導向名稱，還是直接沿用未來 Email 掃描的路徑命名。
