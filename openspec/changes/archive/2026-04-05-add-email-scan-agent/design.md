## Context

專案目前已有可從一般文字中偵測明確中文姓名的垂直切片，但產品方向其實是 Email 與文件中的個資掃描。現況的中文姓名偵測已驗證 Laravel AI SDK、Ollama 與結構化輸出的基本可行性，下一步需要把能力入口提升到 `EmailScanAgent`，同時避免在第一版就把多種個資類型、Tool 編排與多 Agent 流程一次引入，導致範圍過大。

## Goals / Non-Goals

**Goals:**
- 建立 `EmailScanAgent` 作為 Email 文字掃描的第一個能力入口。
- 讓第一版 `EmailScanAgent` 只回傳中文姓名偵測結果，並沿用既有 precision-first 行為。
- 讓 Email 掃描的回應契約可在後續逐步擴充其他個資類型，而不需要重做入口。
- 讓現有中文姓名偵測邏輯可以被 Email 掃描重用，降低重寫風險。

**Non-Goals:**
- 第一版不新增身份證字號、電話、統編或其他個資類型偵測。
- 第一版不引入多 Tool tool-calling 或多 Agent workflow 編排。
- 第一版不處理 Email 標頭解析、附件、OCR 或文件上傳。
- 第一版不要求移除既有中文姓名偵測能力或立即廢棄其 API。

## Decisions

### 1. 以 `EmailScanAgent` 作為新的掃描入口邊界
系統將新增一個專門處理 Email 文字掃描的 Agent，作為後續所有 Email 個資辨識能力的主入口。第一版雖然只輸出姓名結果，但入口名稱先對齊真實產品語意，可避免後續再從單一姓名偵測 API 向外擴張時產生命名與責任混亂。

替代方案：繼續以 `ChineseNameDetectionAgent` 作為主要入口，之後再額外包一層 Email 掃描。此作法短期較省事，但會讓未來的 Email 掃描能力被迫依附在姓名專用能力之下。

### 2. 第一版沿用既有中文姓名偵測語意作為掃描切片
`EmailScanAgent` 第一版只處理從 Email 文字內容中擷取明確中文姓名，並保留現有的 precision-first、evidence 與去重正規化行為。這可讓新入口先建立穩定契約，再逐步加入其他個資類型，而不是同時開多條辨識能力線。

替代方案：第一版同步加入電話、身份證或統編偵測。此作法更接近最終目標，但會大幅增加 prompt、schema、測試與規則驗證複雜度。

### 3. 回應契約在 v1 僅公開 `names`，未來以加欄位方式擴充
第一版 Email 掃描回應將採穩定物件結構，先只公開 `names` 集合。未來若加入 `phone_numbers`、`national_ids` 或其他欄位，採向後相容的欄位新增方式擴充，而不是預先輸出一批空集合。這能避免過早承諾大量尚未定義清楚的欄位語意。

替代方案：第一版就輸出所有預期欄位的空陣列。這能提早塑形最終 schema，但也會讓未來需求尚未定義時先背負不必要契約。

### 4. 先重用既有姓名偵測與 Laravel 後處理模式，不在 v1 強制導入 Tool 編排
雖然 `EmailScanAgent` 長期很適合演進為帶有 Tools 的 orchestration agent，但第一版應優先重用已存在的姓名偵測規則與正常化邏輯，讓變更聚焦在入口遷移與能力收編，而不是同時重構成完整 tool-calling 流程。

替代方案：第一版就讓 `EmailScanAgent` 透過 `tools()` 呼叫姓名清理或其他驗證工具。這較貼近 Laravel AI SDK 的完整設計語感，但目前只做姓名切片時收益有限，且會把風險混在同一個變更中。

### 5. 以增量方式引入 Email 掃描 API，避免一次性破壞既有能力
此變更應新增或切換到 Email 掃描導向的 API 入口，但不要求同時刪除既有中文姓名偵測路由。這可保留比對、回歸測試與漸進遷移空間，等 Email 掃描穩定後再決定是否收斂舊入口。

替代方案：直接以 Email 掃描取代既有中文姓名偵測 API。此作法較乾淨，但會在第一版就引入不必要的相容性風險。

## Risks / Trade-offs

- [Email 掃描入口先只有姓名，容易讓能力看起來名不符實] → 在 proposal、spec 與 API 文件中明確標示 v1 僅支援 `names`。
- [重用既有姓名偵測邏輯可能讓 Email 場景仍保留一般文字偵測的假設] → 以 Email 樣本補測，確認欄位型內容與署名段落仍符合預期。
- [先不導入 Tool 編排，可能與 Laravel AI SDK 的長期理想架構有落差] → 將 `EmailScanAgent` 定位為未來 orchestration 邊界，待新增第二種個資類型時再評估導入。
- [雙入口並存會增加短期維護成本] → 以測試與命名清楚區分主入口與既有能力，後續再依實際使用情況決定是否收斂。

## Migration Plan

1. 建立 `EmailScanAgent` 的 proposal、spec 與實作任務，明確定義 v1 僅掃描姓名。
2. 在應用層新增 Email 掃描請求驗證、控制器與 API 入口，並讓其重用既有姓名偵測語意與後處理。
3. 補齊新的 feature tests，覆蓋成功偵測、空結果、驗證錯誤與精確度負樣本。
4. 保留既有中文姓名偵測能力作為過渡期對照，待 Email 掃描穩定後再決定是否進一步收斂。

## Open Questions

- 第一版是否要保留既有中文姓名偵測對外 API，還是僅保留內部重用能力。
- `EmailScanAgent` 實作時應直接內嵌姓名偵測邏輯，還是以內部服務 / 子能力方式重用既有邏輯。
- 未來加入第二種個資類型時，是否正式將正常化與驗證步驟提升為 Laravel AI Tools。
